# Report Style Specification

## Purpose

This document defines the visual, structural, and editorial style for release-note reports generated by the GitHub Release Note Intelligence Agent.

The reference style is based on:

- `bosgenesis_mop_creation_agent_release_notes_v0_1_0.pdf`
- Source path: `C:/Users/edutavi/OneDrive - Ericsson/Desktop/Release Notes Agents/bosgenesis_mop_creation_agent_release_notes_v0_1_0.pdf`

The generated reports should preserve the same enterprise release-note feel: polished, compact, evidence-backed, readable by leadership, and useful for engineering review.

## Reference PDF Observations

- The sample report is an 11-page professional release package.
- It uses a cover/dashboard first page with product name, version, generated date, source evidence, and high-level metrics.
- It uses repeated page headers with report name, subtitle, generated date, source line, and page number.
- It combines executive narrative, structured metadata tables, metric tiles, readiness/risk summaries, and validation checklists.
- It is generated with a report-first layout rather than a raw Markdown dump.
- It uses concise business-facing language while preserving technical evidence.

## Report Identity

Every report must clearly show:

- Product or repository name.
- Release name or version.
- Repository URL.
- Target branch, tag, or commit SHA.
- Generated timestamp.
- Data sources used.
- Analysis mode and confidence level when available.

## Page Model

### Cover Page

The cover page should function as an executive dashboard.

Required elements:

- Product or project name.
- Report title.
- Release version or release label.
- Repository source.
- Generated date.
- Key evidence sources.
- High-level metrics such as commits, tags, runtime, package version, test evidence, coverage evidence, and report confidence.
- Short release positioning paragraph.
- Audience and purpose table.

### Standard Pages

All non-cover pages should include:

- Compact page header.
- Report title or product name.
- Subtitle such as `Commercial-grade release notes`.
- Generated date.
- Source/evidence summary line.
- Page number.
- Clear section number and section title.

### Footer

Footer content should be restrained. It may include:

- Page number.
- Classification or visibility label when configured.
- Generated by agent name.

## Section Order

The default report section order is:

1. Cover page.
2. Executive summary.
3. Release and package information.
4. Repository and source evidence summary.
5. Project intent and capability overview.
6. Technology and dependency inventory.
7. Feature inventory.
8. Architecture overview.
9. Diagrams.
10. Interface and contract inventory.
11. Code analytics.
12. Commit history and change analytics.
13. Test and coverage analytics.
14. Deployment and operations posture.
15. Security, safety, and governance notes.
16. Risk and readiness assessment.
17. Validation checklist.
18. Known gaps and human-review items.
19. Next-release recommendations.
20. Evidence appendix.

Sections without evidence must still appear when important, but they must say `Not available from repository evidence` or equivalent.

## Visual Style

### Tone

The report should feel:

- Executive-ready.
- Engineering-credible.
- Commercial-grade.
- Evidence-backed.
- Calm and concise.

The report should not feel:

- Like a raw Git changelog.
- Like a README copy.
- Like an unsupported AI essay.
- Like a verbose internal debug dump.

### Layout

- Prefer dense but readable page composition.
- Use clear section hierarchy.
- Use tables for metadata, dependencies, interfaces, risks, and evidence.
- Use metric tiles for headline facts.
- Use callout panels sparingly for release positioning, risk, or readiness.
- Keep paragraphs short.
- Avoid large empty visual areas.

### Color Theme

The default theme should align with BOS Genesis enterprise documentation:

- Deep magenta or brand accent for primary highlights.
- Dark neutral text.
- Light gray section backgrounds where needed.
- White page background.
- Subtle table borders.
- Soft status colors for risk/readiness: green, amber, red, gray.

Exact colors should be configurable in the report template layer.

Suggested initial palette:

| Token | Purpose | Suggested Value |
|---|---|---|
| `brand_primary` | Main accent | `#E20074` |
| `text_primary` | Main text | `#1F2933` |
| `text_secondary` | Supporting text | `#52606D` |
| `border_subtle` | Table/card borders | `#D9E2EC` |
| `surface_light` | Light panel background | `#F5F7FA` |
| `risk_low` | Low risk/pass | `#2F855A` |
| `risk_medium` | Medium risk/warning | `#B7791F` |
| `risk_high` | High risk/fail | `#C53030` |

## Typography

Initial implementation may use system-safe fonts.

Recommended defaults:

- Headings: `Helvetica`, `Arial`, or equivalent sans serif.
- Body: `Helvetica`, `Arial`, or equivalent sans serif.
- Code and paths: monospace.

Rules:

- Section titles must be easy to scan.
- Body copy should be compact and legible.
- Tables should use smaller text than narrative body copy.
- Long paths and package names must wrap safely.

## Data Presentation

### Metric Tiles

Use metric tiles for:

- Commit count.
- Tag count.
- Package version.
- Runtime version.
- Source file count.
- Test file count.
- Coverage percent when available.
- Risk rating.

Each tile should include:

- Numeric or short value.
- Label.
- Optional evidence indicator.

### Tables

Use tables for:

- Package metadata.
- Dependencies.
- Feature inventory.
- Interface inventory.
- Commit analytics.
- Test and coverage summaries.
- Risk register.
- Evidence appendix.

Rules:

- Keep table headings short.
- Prefer compact values over long paragraphs inside cells.
- Use `Not available` instead of blank cells.
- Include evidence paths or evidence IDs where useful.

### Diagrams

Diagrams should be included as Mermaid source in Markdown and rendered or preserved in HTML/PDF.

Expected diagrams:

- Repository analysis flow.
- C4 context.
- C4 container.
- C4 component.
- Deployment topology when evidence exists.
- Interface relationship map.

If diagram confidence is low, the caption must say that the diagram is inferred.

## Evidence and Traceability

Every major claim should connect to evidence.

Report sections should include:

- Source summary line near the beginning.
- Evidence references in tables or captions.
- Evidence appendix at the end.

Evidence references may be:

- File path.
- Commit SHA.
- Tag name.
- Line range.
- Generated artifact ID.
- Analyzer result ID.

Sensitive values must be redacted before appearing in the report.

## Missing Evidence Language

Use explicit, calm language for missing evidence.

Examples:

- `Coverage artifacts were not found in the repository.`
- `Test execution results were not available from repository files.`
- `Deployment topology is inferred from Docker and Helm files.`
- `No formal HLD or LLD document was found.`

Do not imply validation that was not observed.

## PDF Rendering Requirements

The PDF output should:

- Preserve the cover/dashboard page.
- Preserve headers and page numbers.
- Render tables cleanly across pages.
- Avoid clipped text.
- Avoid orphaned headings.
- Include diagrams or diagram source fallback.
- Include artifact metadata and generation timestamp.

Initial implementation may render PDF from HTML using a controlled renderer. The HTML layout should be designed for print from the beginning.

## Markdown Requirements

Markdown output should:

- Be readable without PDF styling.
- Preserve the same section order.
- Include Mermaid diagram blocks.
- Include evidence references.
- Avoid HTML-only meaning.

## HTML Requirements

HTML output should:

- Use the same theme tokens as PDF.
- Be print-ready.
- Preserve tables, metric tiles, callouts, and diagrams.
- Avoid external network assets.

## Acceptance Criteria

- Generated report follows the required section order.
- Cover page includes repository, version/ref, generated date, source evidence, and headline metrics.
- Report contains explicit missing-evidence statements where needed.
- Major claims include evidence references.
- Markdown, HTML, and PDF outputs share the same content model.
- PDF visual style is recognizably aligned with the sample release-note PDF.
